# ⚠️ 测试驱动开发（TDD）检查清单

## 强制规则

**每次添加新功能或修复bug，必须遵循：**

### 红灯-绿灯-重构循环
1. **红灯** - 先写失败的测试
2. **绿灯** - 写最少的代码让测试通过
3. **重构** - 优化代码，确保测试仍然通过

---

## 开发流程检查清单

### 开始开发前
- [ ] 我是否理解了需求？
- [ ] 我是否想到了边界情况？
- [ ] 我是否先写了测试？

### 编写代码时
- [ ] 测试是否覆盖了主要场景？
- [ ] 测试是否覆盖了边界情况？
- [ ] 测试是否覆盖了错误处理？

### 提交代码前
- [ ] 所有测试是否通过？
- [ ] 是否有新增的测试？
- [ ] 测试覆盖率是否足够？

---

## 常见错误检查清单

### API 相关
- [ ] 返回的数据结构是否完整？
- [ ] 必需字段是否都有？
- [ ] 数据类型是否正确？
- [ ] 日期字段是否正确序列化？
- [ ] 错误处理是否完善？

### 前端相关
- [ ] 组件是否能处理 undefined/null？
- [ ] 是否正确处理 API 返回的数据类型？
- [ ] 是否处理了加载状态？
- [ ] 是否处理了错误状态？

### 集成相关
- [ ] 前后端数据格式是否一致？
- [ ] API 返回的字段名是否正确？
- [ ] 日期时间格式是否统一？

## 过去的错误记录

### 2026-03-15 - timestamp 错误
**问题：** API 返回的 messages 没有 timestamp 字段，导致前端 toLocaleTimeString() 报错

**原因：**
1. ❌ 没有遵循 TDD
2. ❌ 没有写 API 测试
3. ❌ 没有检查返回数据结构
4. ❌ 前后端数据格式不一致

**教训：**
1. ✅ 必须先写测试
2. ✅ API 返回的数据结构必须有完整的测试
3. ✅ 前端必须处理可能为 undefined 的字段
4. ✅ 集成测试很重要

**修复：**
- Orchestrator 返回时包含 timestamp
- Chat Page 接收时转换 timestamp 为 Date 对象
- 显示时做保护性检查

## 测试命令

```bash
# 运行所有测试
npm test

# 运行特定测试文件
npm test path/to/test.ts

# 运行测试并生成覆盖率报告
npm run test:coverage

# 监听模式（开发时推荐）
npm run test:watch
```

## 🤖 Agent 角色定义

### 1. PM Agent (产品经理)

**职责：**
- 分析用户需求
- 拆分成可执行的子任务
- 分配任务给合适的 Agent
- 协调团队进度

**输入：**
- 用户的需求描述

**输出：**
- 任务列表（每个任务包含：标题、描述、负责人、依赖关系）
- 需求分析报告

**示例：**
```
用户输入: "帮我创建一个登录页面"

PM Agent 输出:
✓ 已分析需求
✓ 拆分为 3 个子任务：

1. 创建登录表单组件 (Dev Agent)
   - 包含邮箱、密码输入
   - 添加表单验证
   
2. 实现登录 API (Dev Agent)
   - 依赖：任务1
   - 验证用户凭证
   - 返回 JWT token
   
3. 编写测试用例 (Dev Agent)
   - 依赖：任务1、任务2
   - 单元测试
   - 集成测试
```

---

### 2. Dev Agent (开发者)

**职责：**
- 理解任务需求
- 生成/修改代码
- 确保代码可运行
- 提交给 Review Agent

**输入：**
- 任务描述
- 相关文件路径
- 上下文信息

**输出：**
- 生成的代码文件
- 实现说明

**示例：**
```
任务: 创建登录表单组件

Dev Agent 输出:
✓ 已创建文件: src/components/LoginForm.tsx

代码内容:
- React 函数组件
- 表单状态管理 (useState)
- 输入验证
- 提交处理

下一步: 提交给 Review Agent 审查
```

---

### 3. Review Agent (代码审查)

**职责：**
- 检查代码质量
- 安全性审查
- 性能优化建议
- 批准或要求修改

**输入：**
- Dev Agent 提交的代码
- 任务要求

**输出：**
- 审查报告
- 批准 / 需要修改

**审查清单：**
```
✅ 代码风格
✅ TypeScript 类型安全
✅ 错误处理
✅ 可访问性 (a11y)
✅ 性能优化
✅ 安全性检查
✅ 测试覆盖
```

## 📊 任务状态流转

```
pending (待执行)
    │
    │ Agent 开始执行
    ▼
in_progress (执行中)
    │
    │ Dev Agent 完成
    ▼
review (审查中)
    │
    ├─ Review 通过 ────────▶ done (完成) ✅
    │
    └─ Review 需要修改 ────▶ pending (重新执行) 🔄
```

## 🎯 关键设计决策

### 1. 为什么用 Orchestrator？

**问题：** 直接让 Agent 互相调用会混乱
**解决：** 用 Orchestrator 统一调度

**好处：**
- 清晰的职责分离
- 容易追踪和调试
- 可以添加日志、监控
- 方便扩展（加新 Agent）

---

### 2. 为什么每个任务都要 Review？

**问题：** AI 生成的代码可能有 bug
**解决：** Review Agent 检查质量

**好处：**
- 提高代码质量
- 模拟真实团队流程
- 可以学习改进

---

### 3. 为什么保存到文件系统？

**问题：** 代码只在内存里没用
**解决：** 真实保存到文件

**好处：**
- 用户可以直接使用
- 可以用 Git 追踪
- 可以运行测试

## 📝 总结

### 核心流程

```
用户需求 → PM 分析 → Dev 实现 → Review 审查 → 完成
```

### 三层架构

1. **界面层** - 用户交互
2. **协调层** - Orchestrator + Managers
3. **执行层** - Agents + File System

### 设计原则

- ✅ 职责分离
- ✅ 状态清晰
- ✅ 可追踪
- ✅ 可扩展
